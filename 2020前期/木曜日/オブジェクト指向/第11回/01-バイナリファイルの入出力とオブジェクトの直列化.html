バイナリファイルの読み書き

ファイルには、テキスト形式で書き込まれたファイルの他に、2進数の01で表現されるバイナリファイルがあります。

テキストファイルもバイナリファイルの内の一つですが、扱うデータが文字かバイナリかで処理方法が大きく異なります。

バイナリファイルを扱うストリーム

テキストファイルを扱うストリームとして、代表的なものにFileReaderやFileWriterがあります。

それに対して、バイナリファイルを扱うストリームは、代表的なものにFileInputStream、FileOutputStreamがあります。

FileInputStream,FileOutputStream

コンストラクタやメソッドは、FileReaderやFileWriterとよく似ていますが、取り扱うデータが文字からバイト配列に変わっています。

また、write()メソッドや、read()メソッドの読み書きの単位は、1バイト(8bit)ずつとなっています。

バイナリファイル操作系サンプルソース

ファイルのバイナリデータを一括でbyteの配列に格納する

//import java.io.IOException;
//import java.io.UncheckedIOException;
//import java.nio.file.Files;
//import java.nio.file.Path;
byte[] bytes;
try {
    bytes = Files.readAllBytes(Path.of("path/to/file"));
} catch (IOException e) {
    throw new UncheckedIOException(e);
}

ストリームを使った順次取り出し

//import java.io.BufferedInputStream;
//import java.io.IOException;
//import java.io.UncheckedIOException;
//import java.nio.file.Files;
//import java.nio.file.Path;
try (BufferedInputStream bis = new BufferedInputStream(Files.newInputStream(Path.of("path/to/file")))) {
    byte[] b = new byte[1024];
    int index = 0;
    while ((index = bis.read(b, 0, b.length)) != -1) {
        // byte配列には、ファイルの先頭から1024バイトずつ取り出される
        // 一度に取り出すバイト数を変える場合は、byte[]のサイズを変えればよい
        // indexには、取り出されたファイルのインデックスが入り、-1でファイル末尾を表す
    }
} catch (IOException e) {
    throw new UncheckedIOException(e);
}

FileChannel,ByteBufferクラスを使った高速な順次取り出し

FileChannel,ByteBufferを用いると、ストリームを使用した順次アクセスより高速にファイルからデータを取り出せます。

しかしストリームと比べると、覚えなければいけないメソッドが多いため、使う場合は事前に使い方をよく熟知しておくことをおすすめします。

  • FileChannel (Java Platform SE 8 )
  • ByteBuffer (Java Platform SE 8 )
//import java.io.IOException;
//import java.io.UncheckedIOException;
//import java.nio.ByteBuffer;
//import java.nio.channels.FileChannel;
//import java.nio.file.Path;
try (FileChannel fc = FileChannel.open(Path.of("path/to/file"))) {
    ByteBuffer buf = ByteBuffer.allocate(1024); // 1024バイトずつ取り出す宣言
    while (true) {
        buf.clear(); // バッファリセット
        if (fc.read(buf) == -1) { //データを読み込む
            break;
        }
        buf.flip(); // バッファの取り出し位置を入れ替え
        // バッファ内の内容最後まで繰り返して取り出す
        while (buf.hasRemaining()) {
            System.out.println((buf.getChar());   // 文字[char型]を指定して取り出し
        }
    }
} catch (IOException e) {
    throw new UncheckedIOException(e);
}

また、書き込みについては省略しますが、FileChannelを使うと書き込みも可能です。

オブジェクトをそのまま読み書き

テキストデータのファイルの読み書きや、バイナリデータの読み書きについては、前回の講義の手法を用いることでできます。
実際には、データをファイルに保存する形式がテキストデータだと、文字列から各データへの変更が必要だったりしてしまいます。それでは、変換処理を書かなければならなくなり、手間も増えますし、バグを発生させてしまう要因になります。

そこで、登場するのが直列化(シリアライズ) です。
インスタンスまるごとバイナリデータに変換し、ファイルに書き出してしまうことが可能です。もちろん、そのファイルから、もとのインスタンスに復元することも可能です。

RPGを作る際のセーブデータなどは、セーブデータを保持するクラスを作成しておき、そのインスタンスをバイナリで保存しておくなどの用途が考えられます。
（課題では、勇者、敵のデータをテキストファイルで、戦闘途中のデータをインスタンスでバイナリデータとしてファイル保存します。）

直列化の基本的な使い方

オブジェクトを直列化に対応させるには、クラスにjava.io.Serializableインターフェイスを実装する必要があります。

例えば、仮に勇者クラスを直列化に対応させるには以下のようにimplements構文を追加してあげるだけで良いです。

import java.io.Serializable;

public class Hero implements Serializable {
    public Hero(String name, int hp, int atk) {
        this.name = name;
        this.hp = hp;
        this.atk = atk;
    }
    public String name;
    public int hp;
    public int atk;
    public int def;
}

こうしてjava.io.Serializableを実装したクラスは、java.io.ObjectOutputStreamやjava.io.ObjectInputStreamを用いることで、インスタンス自体をストリーム経由でファイルに書き込み、読み込みができるようになります。

では、実際にHeroクラスのインスタンスをファイルに保存し、読み込みを行い別のインスタンスとして生成するサンプルソースを提示します。

//import java.io.IOException;
//import java.io.InvalidClassException;
//import java.io.ObjectInputStream;
//import java.io.ObjectOutputStream;
//import java.nio.file.Files;
//import java.nio.file.Path;
Hero h = new Hero("たろう", 100, 50); //引数はテキトーです。

//書き出し処理
try (ObjectOutputStream oos = new ObjectOutputStream(Files.newOutputStream(Path.of("taro.dat")))) {
  //Object書き出し
  oos.writeObject(h);
  //強制保存
  oos.flush();
} catch (IOException e) {
    throw new UncheckedIOException(e);
}

//読み込み処理(書き込んだデータを流れで読み込みます。)
try (ObjectInputStream ois = new ObjectInputStream(Files.newInputStream(Path.of("taro.dat")))) {
    //ObjectをHeroにキャストして読み込み
    Hero h2 = (Hero)ois.readObject();

    System.out.println(h2.name);
    System.out.println(h2.hp);
    System.out.println(h2.atk);

} catch (ClassNotFoundException | InvalidClassException e) {
    System.out.println("データファイルが壊れている可能性があります。");
} catch (IOException e) {
    throw new UncheckedIOException(e);
}

これでインスタンスをファイルに保存、読み込みできるようになります。

Serializableを実装するということは

単に、インスタンスを直列化できるようにする。という意味合いではありません。

Serializableインターフェイスの中身を見るとわかりますが、メソッドなどの定義はなく、単に目印としてのインターフェイスであることがわかります。
(こういうインターフェイスを「マーカーインターフェイス」なんて呼んだりします。)

これが意味することとは、「このクラスはJavaの仕様に従って直列化しても問題ないクラスです。」 ということを明示的にしているだけであって、「直列化してください」という意味ではないことに注意してください。

直列化の注意点

非常に便利な直列化からのファイル保存ですが、注意点もあります。

例えば、HeroクラスがWeaponクラスのインスタンスをフィールドとして保持している場合、Weaponクラスもjava.io.Serializableを実装していないと、読み込みを行った場合にnullが代入されてしまいます。これでは、本来nullが代入される予定のない変数にnullが入り込んでしまうことになり、その後の処理ではNullPointerExceptionが発生してしまいます。

なので、直列化したいクラスにはすべてjava.io.Serializableを実装しておきましょう。

ただ、Heroクラスがjava.io.Serializableを実装しているWeaponクラスのインスタンスを保持している場合は、Heroクラスのインスタンスを直列化するだけで、Weaponクラスのインスタンスも同時に直列化されます。 個別に直列化して保存する必要がないので便利ですね。

他にも注意点があります。

直列化の際に気を付けなければならない点をまとめると…

  1. java.io.Serializableを実装していないクラス型のフィールドは直列化の対象にならない(プリミティブ型のフィールドは除く)
  2. static修飾子が付いている静的なフィールドは直列化の対象にならない
  3. transient修飾子が付いている直列化対象外となるフィールドは直列化の対象にならない

厄介な点として、1.の直列化の対象にならないクラス型のフィールドが含まれる場合にエラー等は出ず、読み込みを行った際にしれっとnullが代入されてしまうことがあげられます。 直列化の実装時には注意しておきましょう。

シリアルバージョンUID

Heroクラスのインスタンスをファイルに保存したあとに、Heroクラスへ変更が入ってしまった場合、復元されたタイミングで矛盾が発生してしまいます。

この矛盾自体を回避する方法はありません。※矛盾を解消するための旧クラスと新クラスの橋渡しプログラムを作成する場合をのぞく

矛盾が発生している場合に例外を発生させ、例外処理にて代替処理を行う方法として、シリアルバージョンUIDというフィールドを宣言しておくことがあげられます。

public class Hero implements java.io.Serializable {
  //シリアルバージョンUID
  private static final long serialVersionUID = 545154231343L;


}

シリアルバージョンUIDはlong型で、名前、修飾子ともに揃えておきましょう。 この時、値は適当で良いですが、クラスの仕様が変更された場合には必ず値の変更を忘れないようにしましょう。

こうしておくと、クラスに仕様変更があった場合、それ以前に保存されたインスタンスを復元しようとしたタイミングで例外(java.io.InvalidClassException)が発生するようになります。 これで矛盾を残したままインスタンスが復元されることをなくせます。

また、シリアルバージョンUIDを宣言しない場合、JavaVMによって自動的に生成されたシリアルバージョンUIDが内部的に使用されます。

これをあえて明示的に書く理由は、Javaのバージョンアップの際や、異なる環境でのコンパイルによってシリアルバージョンUIDに差異生じる場合があります。 内部的なデータはファイルを移動することによってそのまま対応が可能なはずですが、自動生成されたシリアルバージョンUIDに差異が生じてしまうことにより意図しない例外(java.io.InvalidClassException)が発生してしまいます。 これを回避するためにも、直列化が必要なクラスにはシリアルバージョンUIDは設定するように心がけましょう。

ファイル処理まとめ

前回の講義内容を含め、紹介したファイル処理を使うことで、プログラム中のデータを「永続化」することが可能です。

単純なテキストファイル処理や、最後の直列化などは、扱い方次第で非常に便利です。

「永続化」の手法については他にもデータベースを利用する手法などもありますが、単純なデータを単純な仕組みで動作させるには、やはりファイル処理が良いでしょう。 データベースについては、後期の講義で簡単に触れたいと思っています。
