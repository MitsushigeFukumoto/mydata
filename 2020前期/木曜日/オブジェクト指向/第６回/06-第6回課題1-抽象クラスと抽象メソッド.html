第6回課題1 - 抽象クラス・抽象メソッド

オブジェクト指向プログラミングおよび演習1 第6回

抽象クラスについて学んだことを活かし、SimpleRPGをもっと扱いやすいものにブラッシュアップしましょう

1. CharacterBaseクラスの抽象クラス化

前回の課題で作成したHero,Enemy両方の親クラスであるCharacterBaseクラスは、それ自体が親クラスとしての役割しかありません。

このクラスを個別にインスタンス化することはないため、このクラスを抽象クラスに変更しましょう。

2. CharacterBaseクラスのattackメソッドを抽象メソッド化

CharacterBaseを継承したHero,Enemy両方のクラスでの主な違いは、Heroクラスにてオーバーライドしている攻撃を行う処理int attack(CharacterBase e)です。Enemy側にも何らかの処理を加えたいと思っているため、このメソッドは親クラスで処理を決定させずに、継承を行った2つのクラスにて詳しい処理内容が決定されるよう抽象メソッドに変えましょう。

加えて、前回の課題にて、戻り値がintであることが原因でクリティカルヒットの判定ができない旨をお話しました。今回は、この点も改善することを考えながら、いくつかの変更を行っていきます。

2-1. AttackResultクラスの作成

このクラスは、attackメソッドの戻り値の型として使用されます。

クラスのソースコードは以下の通りです。

package jp.ac.ait.k00000;

public class AttackResult {
    boolean isFailure = false;  // 攻撃失敗時true
    boolean isCritical = false; // クリティカルヒット時true
    int damage = 0;             // 与えたダメージ量
}

2-2. CharacterBaseクラスのattackメソッドを変更

現在のCharacterBaseのint attack(CharacterBase e)メソッドを削除し、以下の抽象メソッドに置き換えます。

public abstract AttackResult attack(CharacterBase e);

これにより、Hero, Enemy, SimpleRPGにてコンパイルエラーが発生するようになります。
抽象メソッド化に伴う修正を行っていきます。

2-3. Heroクラスにpublic AttackResult attack(CharacterBase e)を実装する

抽象メソッドの中身をHeroクラスにて実装します。

内容は、以前のHeroクラスのattackメソッドと同じで良いのですが、戻り値がAttackResultになっているため、以下の条件にしたがい、戻り値を作成して返すようにしてください。

  1. 攻撃失敗時
    • isFailure に true を設定
    • damage に 0 を設定
  2. クリティカルヒット成功時
    • isCritical に true を設定
    • damage は計算されたダメージ値を倍にして設定
  3. 通常攻撃時
    • damage は計算されたダメージ値をそのまま設定

2-4. Enemyクラスにもpublic AttackResult attack(CharacterBase e)を実装する

CharacterBaseのattackメソッドを抽象メソッドにしたため、Enemyクラスでもattackメソッドを実装する必要があります。

ついでなので、敵キャラクターの攻撃時に、20%の確率でクリティカルヒットするように機能追加を行います。 (攻撃成功率はいままでどおり100%でOKです。)

attackメソッドの中身は、Heroクラスより必要に応じて抜粋して持ってくると良いでしょう。

2-5. SimpleRPGにてattackメソッドを呼んでいる箇所の修正

戻り値の型がAttackResultになり、攻撃を行った場合の詳細情報が取得できるようになりました。

SimpleRPGのmainメソッドにて、勇者、敵それぞれの攻撃行動を行っている箇所をそれに合わせて修正しましょう。

  1. 攻撃を失敗した際は、isFailureがtrueになっている
  2. クリティカルヒット時、isCriticalがtrueになっている
  3. 与えたダメージ値は、damageに入っている

上記3点を使い、クリティカルヒットのときのメッセージを変更するなどしてください。

2−5ex. SimpleRPGのメインメソッドをブラッシュアップ

これは任意です。

勇者の行動メソッド(行動選択から行動、ゲームオーバー判定までを行うメソッド)でprivate static boolean heroAction(Hero h, Enemy e)や、敵の行動メソッド(敵の行動、ゲームオーバー判定までを行うメソッド)でprivate static boolean enemyAction(Hero h, Enemy e)などを作成すると、修正箇所を一つに絞れるので修正しやすくなるでしょう。

ちなみに、booleanの戻り値は、ゲームが終了する場合は false を返すようにすると、無限ループを抜けるパターンが用意できるでしょう。
